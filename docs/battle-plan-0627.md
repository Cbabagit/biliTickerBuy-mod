## 时间线

### 12:00 首开

| 时间 | 操作 |
|------|------|
| 11:45 | 确认两个终端的 time-start 都设为 12:00 |
| 11:50 | 启动 LV5 进程 |
| 11:52 | 启动 LV2 进程（隔 2 分钟，避免预热请求重叠） |
| 12:00:00 | 自动开始抢票。**两个进程各自独立抢** |
| 12:00~12:05 | 刚开票时大概率会大量 429（抽奖机制），不要慌！ |
| 12:05~12:10 | 如果还在大量 429，这是正常的。B站刚开票=抽奖 |
| 12:10 | 如果双方都没抢到且票池显示「暂时售罄」→ 进入回流模式 |

### 首开→回流过渡

12:10 之后：
- **如果显示了「售罄」**：不需要改配置，脚本会自动处理
- **保持运行**：回流票不定时放出，不要关程序
- **回流规律**：每天 ~11:30 / ~12:55 / ~14:55 / ~20:55 有集中放票

### 18:00 二开

| 时间 | 操作 |
|------|------|
| 17:30 | 确认脚本仍在运行（不需要重启，回流模式会自动检测二开） |
| 18:00:00 | 二开开始，和首开一样的策略 |
| 18:10 | 如果有票但你没抢到 → 继续回流模式挂机 |

### 回流挂机（6/27 ~ 7/12）

- **不需要关程序**。两个终端窗口一直开着
- 每天 ~11:30 / 12:55 / 14:55 / 20:55 集中放票
- 直到 7/12 开场前，都会有人退票回流
- **关键原则**：回流票遵循 1 秒盾，挂机时间是朋友

## 你可能会看到的错误码对照

| 代码 | 含义 | 处理方法 |
|------|------|---------|
| 429 | 请求太多被限流 | **正常现象**。指数退避会自动等 1s→2s→4s→...→30s |
| 412 | IP 被封 | LV5 412 → LV2 还在抢。LV2 412 → LV5 还在抢 |
| 100001 | 中层限流 | 不影响抢票，继续下一轮 |
| 100009 | 库存不足 | 没票了，等回流 |
| 100041 | 抢了未开售的场次 | 换账号 or 等解封（约5min） |
| 100051 | token 过期 | 自动重新 prepare |
| 0 | **抢票成功** | 快去付款！！！ |

## 抢到票后的操作

1. 终端会弹出付款链接或二维码
2. 浏览器自动打开支付页面
3. 手动扫码 / 登录 B 站完成支付
4. **B 站的订单 15 分钟不支付自动释放**
5. 如果不需要这张票了，可以放弃支付，票会回流给其他人

## 两种模式的区别

### 首开模式（interval=500ms + jitter）
- 大量 429 = 抽奖入场券
- 1 秒盾期间每秒只会有 1 次有效请求
- 快速试错：500ms 间隔 × jitter = 平均 ~650ms 一次
- 同账号 2 进程：共享状态文件会让它们交替发请求，不会撞 1 秒盾

### 回流模式（interval=1000ms + jitter）
- 429 很少或没有
- 1 秒盾是主要限制
- 1000ms 间隔严格对齐 1 秒盾 → 每次请求都能落在有效窗口
- 不拼速度，拼耐心

## 💡 最重要的提醒

> **不要用自己的生产账号去试非目标场次**，100041 会封会员购 5 分钟。
> 6/27 当天不要打开任何 B 站会员购页面以外的页面，保持 cookie 干净。
> 如果某个进程 412 了，**不要杀掉它**——它冷却完会自动恢复。
> 两个进程互相独立：一个 412 不影响另一个。
